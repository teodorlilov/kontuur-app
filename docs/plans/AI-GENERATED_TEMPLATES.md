Plan: AI-Generated, Brand-Consistent Post Templates (Kontuur)
Context
Today Kontuur generates post text with Claude, but post visuals are entirely manual: users upload a PNG or import a Canva export (image-slot.tsx). There is no way for a client to have on-brand post imagery generated for them.

Goal: each client gets a Brand Kit (design system: colors, fonts, logo, style) and a library of AI-generated HTML/CSS templates constrained to that kit. Templates are generated at onboarding / on-demand (not per-post), human-approved once, then reused across posts. At post time Claude fills a template's text slots, we render a PNG, and it flows through the existing image → post_images → Instagram auto-post pipeline. Templates can render over a background photo generated by a separate AI image model (Claude designs the frame + writes the copy; the image model produces the photo).

Why this shape: variety comes across clients (each kit is unique) while consistency holds within a client (posts reuse the same approved templates → coherent feed). The AI is creative on layout only — colors/fonts come from tokens as CSS variables; raw literals are lint-rejected, so brand drift is structurally impossible.

This plan respects docs/CLAUDE.md: reuse before adding, single source of truth, ≤40-line single-purpose functions, no any, no unrequested refactors, no regressions/bottlenecks.

Architectural decisions
Renderer: satori + satori-html + @resvg/resvg-js (only new runtime dependency). Pure-Node (no Chromium), low cold-start, deterministic. Its flexbox-only CSS subset is a feature: it prevents the AI emitting CSS we cannot validate. Puppeteer rejected (≈300 MB Chromium, serverless-hostile, full-CSS surface defeats token enforcement). App is Node runtime, Next 16 App Router — no edge constraints.
Vision via a backward-compatible extension to callAnthropic — add one optional images field; the existing string path is left byte-for-byte unchanged.
Extract two shared primitives (mandated by the "extract before a third copy" rule, not gold-plating): a generic uploadImage core from uploadPostImage, and runInBatches from the generation orchestrator's batch loop.
New tokens are genuinely new data. brand_profiles holds semantic brand (tone, audience, pillars) — it has no visual tokens. So a new design_systems table (one-to-one with clients, mirroring brand_profiles) is correct, not duplication.
Which tool produces what: Claude designs the template (HTML/CSS) and writes the copy; satori+resvg rasterizes that to the final PNG (deterministic, not a model); a separate generative image model — fal.ai Flux (synchronous REST), behind a swappable image-provider interface — produces only the optional background photo. Claude cannot generate photos.

Key technical designs
Vision extension (src/utils/ai-client.ts)
Add one optional field to CallAnthropicOptions; change only the message-build branch:

images?: Array<{ base64: string; mediaType: 'image/png' | 'image/jpeg' }>
No images → content: userMessage (string) — unchanged request.
With images → content: [...imageBlocks, { type:'text', text:userMessage }] via a ≤10-line toImageBlock helper. Throw early if images + assistantPrefill (like outputSchema already does). images IS compatible with outputSchema. Streaming/retry untouched. The wholesale mock (mocks/ai-client.ts) needs no change.
Rendering & caching
injectTokenVariables → satori-html → satori(tree,{width,height,fonts}) → Resvg(...).render().asPng(). Fonts fetched from Google Fonts, cached in a module-level Map, with a bundled Inter fallback so a network blip degrades instead of failing. After render, upload the thumbnail via the new uploadImage to ${clientId}/templates/${templateId}.png and store thumbnail_url/thumbnail_storage_path on the row — the library UI reads the cached URL and never re-renders for display. Rendering stays off per-request hot paths (generation is on-demand); batch renders bounded by runInBatches; maxDuration set.

Data shapes
callAnthropic template outputSchema: { name, html, css, aspect:'square'|'portrait', fields_schema:[{ key, label, max_chars?, role? }] } (required: name/html/css/aspect/fields_schema). Parse with extractToolInput<GeneratedTemplate>() (ai.ts).

Data model
design_systems (1-1 clients): id, client_id unique, colors jsonb, fonts jsonb, logo_url, logo_storage_path, style_prefs jsonb, source text, created_at, updated_at.
templates: id, client_id, name, html, css, fields_schema jsonb, aspect, status default 'pending_approval', thumbnail_url, thumbnail_storage_path, validation_json, timestamps; index (client_id,status).
template_generations (audit): id, client_id, template_id, pass_number, critique_json, validation_json, repaired bool, created_at; index (client_id,created_at desc).
RLS enabled on all three; admin client for writes, server (RLS) client for reads; ownership via verifyClientOwnership() (helpers.ts).

Regression / performance risks & de-risking
callAnthropic extension — string path left literally unchanged; add a unit test asserting content is still a string when images absent; tsc --noEmit (widely imported).
uploadImage extraction — uploadPostImage keeps identical path + return shape; verify the images route still returns same publicUrl/storagePath.
runInBatches extraction — verbatim loop, same chunk size + error logging; run existing generation tests; confirm identical settled handling.
satori/resvg dependency — cold-start/memory: keep off hot paths, module-cache fonts, bound batches, bundle fallback font, measure render time before wiring into auto-post.
IG media scope — instagram_business_basic (already requested in meta/connect) should already permit GET /{ig-user-id}/media for the own account; verify with a live call before building extraction — do NOT add a scope or force re-auth unless a real permissions error occurs.
AI image model (M6) — cost/latency (seconds, paid per image) and unpredictable/off-brand output. De-risk: behind a swappable image-provider interface (no lock-in); run only at compose/approval time, cache results, bound via runInBatches, maxDuration; style-anchored prompts + reuse M3 vision critique to reject and regenerate bad backgrounds; new env var documented in .env.example. No native compositing dep (data-URI layer in satori).
Strategic risks & spike-first
Beyond the per-change regressions above, these are the risks most likely to derail the effort or cause an incident:

Satori fidelity vs AI CSS (highest technical) — satori renders only a CSS subset and will silently mis-render arbitrary AI HTML/CSS. The validate→repair loop must absorb a real failure rate; hard prompt constraints + lint are necessary but not sufficient.
Non-Latin font coverage — multi-language app (secondary_language, Cyrillic locale). Satori needs the exact font buffer AND glyph coverage, or text renders as tofu (□). Brand fonts often lack Cyrillic/Greek/Arabic. Plan font fallback per script.
@resvg/resvg-js native binary on the deploy target — must load on the production host (Vercel/serverless bundling, serverExternalPackages, cold-start). Blocks the whole renderer if not.
Auto-posting AI backgrounds unreviewed (highest product risk) — M6 backgrounds are generated at compose time; combined with autonomous generation + auto-post, a bad image could reach a real account. Require approval before AI-background posts auto-publish (or keep behind the autonomous pause) until the pipeline is trusted.
Taste is not code-verifiable — validators catch broken, not bland; the human approval gate means this is assisted, not fully autonomous.
Cost & latency — stacked Claude + vision + fal calls per pack; needs progress UX + budgeting.
Feed consistency depends on UX — nudge clients toward a small active template set.
Spike first (do this inside M1, before M2–M6): render a real AI-generated template (not the seed) on the actual deploy target, in Cyrillic + Latin, and measure the render-failure rate. If satori fidelity or the resvg binary is a problem, this surfaces it in ~2 days instead of after the whole build.
